<completed_work>

- **Fix 1 — Two-instance removal crash** (PluginEditor.cpp + PluginProcessor.cpp):
  Root cause: `onConnectionChangeUI` lambda captured `this` (PluginEditor). When editor was deleted first, destructor did NOT clear the callback. Later, `~GamepadInput()` fired the stale lambda → `callAsync` with dangling pointer → crash.
  Fix: `~PluginEditor()` now clears `proc_.getGamepad().onConnectionChangeUI = nullptr` before any other teardown. `~PluginProcessor()` clears `gamepad_.onConnectionChange = nullptr`.
  Debug file: `.planning/debug/two-instance-crash-and-track-type.md`

- **Fix 2 — Ableton freeze at audio init** (GamepadInput.cpp + GamepadInput.h):
  Root cause: SDL background init thread (spawned immediately at plugin construction) ran `SDL_Init` concurrently with Ableton's WASAPI/ASIO audio driver init → HID device lock contention / COM apartment deadlock.
  Fix: SDL background thread now spawns only after a 4-second delay from construction (measured in timerCallback via `constructTime_` and `kSdlInitDelayMs = 4000`). This ensures SDL_Init happens well after Ableton's audio engine is settled.
  Debug file: `.planning/debug/ableton-freeze-audio-init.md`

- **Fix 3 — Track type change to audio** (NOT a code bug):
  Stale Ableton plugin cache from old installed version. Resolved by build + reinstall.

- **Fix 4 — MIDI channel dropdown not grayed out**:
  The plugin's own `voiceChBox_` dropdowns ARE correctly toggled by `singleChanMode`. The dropdown the user saw was Ableton's own MIDI routing UI (plugin can't control that). No fix needed.

- **Previous session fixes** (GamepadInput.cpp + PluginProcessor.cpp — gamepad reconnect crash + MIDI silence):
  Also in working tree, built and installed this session.

- **Build + install**: Done successfully via temp script `__install_tmp.ps1` (now deleted).
  Installed to both `%LOCALAPPDATA%\Programs\Common\VST3` and `C:\Program Files\Common Files\VST3`.
