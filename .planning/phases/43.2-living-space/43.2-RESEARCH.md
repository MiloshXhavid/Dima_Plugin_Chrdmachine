## User Constraints (from CONTEXT.md)

### Locked Decisions

**Star twinkle**
- Each star flickers independently via slow sinusoidal oscillation + LCG random phase offset per star
- Amplitude ±8–12% of base alpha; frequency 0.3–0.8 Hz per star (never synchronized)
- Brightness only — no color shift

**Shooting stars / micrometeorites**
- One streak every ~20s average (random interval 12–30s)
- Short fast streak 40–80px long, 1–1.5px wide, white with alpha fade-in/out over ~0.3s total
- Random angle and position, peak alpha ~0.5

**Parallax depth layers + unified flow direction**
- Replace random per-star direction with unified flow model: all stars share dominant drift direction, per-star variation ±15°
- Joystick deflection ROTATES drift heading over time (accumulated heading, not pointing)
- Heading persists on joystick release — no rubber-band snap; full 360° rotation possible
- Turn rate: ~15°/second at full deflection (~0.004 rad/tick at 60 Hz)
- Read deflection from `displayCx_/displayCy_` relative to pad center
- New member: `float driftHeading_` initialized to -π/2 (upward/forward)
- Parallax: Near (r > 1.4) = 100% speed, Mid (r 0.8–1.4) = 60%, Far (r < 0.8) = 30%

**Star respawn with fade-in (replaces torus wrap)**
- Stars exiting any edge respawn at random (x, y) anywhere within pad
- Fade in over ~0.5s (~30 ticks at 60 Hz)
- Per-star counter: `uint8_t fadeIn` (0 = fully visible, counts down from 30 on respawn)
- Replaces existing 4-line torus wrap entirely

**Nebula gas clouds**
- 2–3 soft elliptical blobs, purple/teal palette, alpha ~0.04–0.07
- Radius 80–180px, radial gradient fill
- Full pad crossing in ~90–120s, torus wrap
- Rendered below the starfield layer

**Background hue drift**
- Base background fill (`Clr::bg` = `0xFF131525`) shifts hue ±8° over 120s cycle
- Computed live in `paint()` via `sin(Time::getMillisecondCounterHiRes() / 120000.0 * twoPi) * 8.0`

**Warp mode compatibility**
- Twinkle: multiply twinkle offset by `(1.0f - warpT)` when warp present
- Shooting stars: no new spawns when `warpT > 0`; fade in-flight streak with `(1.0f - warpT)`
- Parallax: overridden by Phase 42 freeze rules during warp (r > 1.2 freeze threshold is Phase 42's, not Phase 43.2's)
- Nebula clouds: fade alpha with `(1.0f - warpT)`
- Hue drift: continues during warp — no interaction

None specified in CONTEXT.md.

- No cockpit/HUD overlay
- No cursor breathing or gravitational lensing
- No new APVTS parameters

## Summary

Phase 43.2 adds six layered ambient animation effects to the JoystickPad that make idle (non-warp) space feel alive: unified drift heading with joystick steering, per-star twinkle, shooting star streaks, parallax depth stratification, nebula gas clouds, and a subliminal background hue shift. All effects are paint-only additions to the existing JoystickPad component — no new APVTS parameters, no audio-thread changes.

**Critical discovery — Phase 42 was reverted.** The git log confirms all Phase 42 commits (warp effect) were reverted. The current codebase has NO `warpRamp_`, `warpT_`, `WarpStar`, or `warpStars_` members. The CONTEXT.md references `warpT` for crossfade compatibility. The planner must resolve this dependency: either Phase 43.2 is planned WITH a Phase 42 re-implementation preceding it, or Phase 43.2 must make `warpT` crossfades conditional on a `warpRamp_` member that may not exist and default-safe (i.e., treat warpRamp_ == 0.0f as the always-idle case until Phase 42 is re-implemented).

**Primary recommendation:** Plan Phase 43.2 as a self-contained change that adds all living-space effects with warp crossfade guards written as `const float warpT = warpRamp_;` where `warpRamp_` will be added by Phase 42 — but also add `warpRamp_ = 0.0f` as a new member in Phase 43.2's header work if Phase 42 is not re-done first. This makes Phase 43.2 safe to ship standalone and forward-compatible with Phase 42.

The `StarDot` struct currently lacks `twinklePhase`, `baseAngle`, and `fadeIn` fields. All three must be added. The existing 4-line torus wrap in `timerCallback` is fully replaced by the respawn-with-fade-in logic.

**Primary recommendation:** Two-plan wave structure: Plan 01 extends `StarDot` + header additions + timerCallback logic (stars, heading, nebulae state), Plan 02 adds `paint()` draw passes (nebulae, hue shift, updated star draw with twinkle + fade-in, shooting star draw).

## Architecture Patterns

### Recommended Structure

The work touches exactly three methods of `JoystickPad`:

```
Source/PluginEditor.h   — StarDot struct extension + new member declarations
Source/PluginEditor.cpp
  JoystickPad::resized()         — nebula init (scatter positions/angles)
  JoystickPad::timerCallback()   — all per-tick animation state updates
  JoystickPad::paint()           — all draw passes
```

### Pattern 1: StarDot Struct Extension

**What:** Add three fields to the existing `StarDot` struct.
**Current struct (PluginEditor.h line 136):**
```cpp
struct StarDot { float x, y, r, vx, vy; juce::Colour c; };
```
**New struct:**
```cpp
struct StarDot {
    float x, y, r, vx, vy;
    juce::Colour c;
    float twinklePhase = 0.0f;   // 0..2π, per-star sine phase offset
    float baseAngle    = 0.0f;   // dominant drift direction ± 15° variation
    uint8_t fadeIn     = 0;      // counts down from 30 on respawn (0 = fully visible)
};
```
**When to use:** Struct is declared in JoystickPad private section. Field defaults mean no change to existing construction sites — resized() regeneration will set them explicitly.

**Important:** `vx/vy` are now derived quantities (computed from `driftHeading_ + baseAngle` + depth multiplier each tick) and are no longer stored directly. The resized() generation code that currently bakes `vx/vy` from a random angle must be changed to store `baseAngle` instead, with `vx/vy` fields retained for backward compatibility (or removed and computed inline). The simplest approach: **keep vx/vy fields but ignore them** — compute motion inline in timerCallback from `baseAngle` + `driftHeading_`. This avoids touching the paint() starfield draw loop which reads only `x, y, r, c`.

### Pattern 3: timerCallback Animation Block Order

Insert new blocks in timerCallback in this order (before the final `repaint()` call):

1. **Drift heading update** — read joystick deflection from `displayCx_/displayCy_`, update `driftHeading_`
2. **Starfield position update** — replaces existing torus-wrap loop; uses `effectiveAngle = driftHeading_ + s.baseAngle`, depth multiplier, respawn logic
3. **Twinkle phase advance** — `s.twinklePhase += twinkleFreq_per_star * dt` (or computed from a global tick counter + per-star phase offset)
4. **Shooting star tick** — advance timer; on interval elapsed, spawn; advance progress; on completion, reset + new random interval
5. **Nebula position update** — extremely slow drift, torus wrap
6. **Existing warp ramp block** — (will come from Phase 42 or already 0.0f)

### Pattern 5: Hue Drift Implementation

```cpp
// In paint(), replacing the plain g.fillRect(b) at Layer 1:
const double ms = juce::Time::getMillisecondCounterHiRes();
const float hueOffset = (float)(std::sin(ms / 120000.0 * juce::MathConstants<double>::twoPi) * 8.0);
const juce::Colour bgBase { 0xFF131525 };
const float baseHue = bgBase.getHue();

```

### Pattern 6: Nebula Radial Gradient Draw

```cpp
// For each nebula blob in paint():
const float px = blob.x * w, py = blob.y * h;
juce::ColourGradient grad(
    blob.colour.withAlpha(blob.alpha * (1.0f - warpT)),  // centre
    px, py,
    blob.colour.withAlpha(0.0f),                          // edge
    px + blob.rx, py,
    true);  // isRadial = true
g.setGradientFill(grad);
// Draw as ellipse path (ColourGradient with isRadial draws circles; use AffineTransform to stretch to rx/ry)
juce::Graphics::ScopedSaveState ss(g);
g.addTransform(juce::AffineTransform::scale(1.0f, blob.ry / blob.rx, px, py));
g.fillEllipse(px - blob.rx, py - blob.rx, blob.rx * 2.0f, blob.rx * 2.0f);
```
**Pitfall:** `juce::ColourGradient` with `isRadial=true` draws a circular gradient — to get an ellipse, scale the Graphics context before drawing. This is the correct JUCE approach (HIGH confidence — verified against JUCE source patterns).

### Pattern 7: Shooting Star Draw

```cpp
// Streak as gradient line (head bright, tail transparent)
const float streakLen = juce::jmap(progress, 0.0f, 0.3f, 0.0f, streakPx)   // fade in
                      * juce::jmap(progress, 0.7f, 1.0f, 1.0f, 0.0f);      // fade out
// Head position
const float hx = startX + std::cos(ss.angle) * progress * totalDistPx;
const float hy = startY + std::sin(ss.angle) * progress * totalDistPx;
// Tail position (behind head)
const float tx = hx - std::cos(ss.angle) * streakLen;
const float ty = hy - std::sin(ss.angle) * streakLen;
const float peakAlpha = 0.5f * (1.0f - warpT);  // fades during warp
juce::ColourGradient sg(
    juce::Colours::white.withAlpha(peakAlpha * 0.15f), tx, ty,  // tail dim
    juce::Colours::white.withAlpha(peakAlpha),          hx, hy,  // head bright
    false);
g.setGradientFill(sg);
g.drawLine(tx, ty, hx, hy, 1.2f);
```

### Anti-Patterns to Avoid

- **Baking nebulae into `spaceBgBaked_`:** The CONTEXT.md explicitly forbids this — nebulae drift and their alpha must be multiplied by `(1.0f - warpT)`, which requires live rendering.
- **Storing `warpT` as a member:** Per project style (confirmed from Phase 42 plan), `warpT` is always computed locally as `warpRamp_ * warpRamp_ * (3.0f - 2.0f * warpRamp_)`. Do not add `warpT_` as a member.
- **Advance twinkle phase in paint():** Never. Only timerCallback advances simulation state. paint() is read-only from the perspective of simulation state.
- **Calling `juce::Random` in paint():** Never — paint() can be called at any time by the OS. All randomness advances in timerCallback only.
- **Using `getMillisecondCounterHiRes()` in timerCallback for animation:** Use tick-counting instead (advance phase by a fixed amount per tick). Reserve `getMillisecondCounterHiRes()` for the hue drift in `paint()` only (as specified in CONTEXT.md), because it is stateless (no drift accumulation needed there).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Elliptical gradient fill | Custom pixel loop | `juce::ColourGradient(isRadial=true)` + `g.addTransform(scale)` | Built-in, hardware accelerated |
| Hue rotation | Manual HSL math | `juce::Colour::fromHSV()` + `getHue()` | JUCE handles wraparound |
| Star position seeding | Time-based rand | `juce::Random(fixedSeed)` in resized() | Deterministic — avoids visual pop on resize |
| Trig per-frame | Lookup table | `std::cos/sin` directly | 250 stars × 1 cos+sin per frame is negligible |

## Common Pitfalls

### Pitfall 1: Phase 42 Dependency on warpT
**What goes wrong:** CONTEXT.md references `warpT` in every crossfade formula, but Phase 42 was reverted — `warpRamp_` does not exist in current code.
**Why it happens:** Phase 42 was planned before 43.2; its revert was not reflected in the 43.2 CONTEXT.md.
**How to avoid:** Add `float warpRamp_ = 0.0f;` as a new JoystickPad member in Phase 43.2's header task. This makes all `(1.0f - warpT)` crossfade guards safe (warpT will always be 0.0f until Phase 42 is re-implemented). When Phase 42 lands, it will overwrite `warpRamp_` with its own ramp logic — no conflict.
**Warning signs:** Compiler error "undeclared identifier warpRamp_" if guard is omitted.

### Pitfall 2: StarDot Struct Change Breaks resized() Generation
**What goes wrong:** Adding fields to `StarDot` doesn't break existing push_back calls (defaulted fields), but the `baseAngle` field needs to be populated and the old `angle → vx/vy` code in resized() must be changed to store `baseAngle` instead.
**Why it happens:** The existing resized() already computes an `angle` and bakes it into `vx/vy`. After this phase, `vx/vy` are recomputed each tick from `driftHeading_ + baseAngle` — the baked values become stale on first tick.
**How to avoid:** In resized(), set `s.baseAngle = angle` (the ±15° variation from upward) instead of `s.vx = cos(angle)*speed; s.vy = sin(angle)*speed`. Keep `speed` (normalized) as a field too — `r` is reused as the radius discriminator for depth layers, not speed. Check if speed is already encoded in `r` or needs a separate field.

**Critical check:** Current `StarDot` has `vx, vy` as velocity components — they serve as both speed AND direction. After this change, direction comes from `driftHeading_ + baseAngle` and the magnitude needs to be preserved. The cleanest approach: keep `vx, vy` as legacy fields but use them to store the per-star speed scalar (set `vx = speed` during generation, use `vx` as speed in timerCallback). Alternatively, repurpose `r` (already used as visual radius) and keep `vx` as the speed scalar. The CONTEXT.md shows `s.speed` in the formula — which does not yet exist as a StarDot field. This needs a decision in the plan.

**Recommended resolution:** Add `float speed` to `StarDot` explicitly. Set during resized() generation. Use `vx` and `vy` for nothing (or remove them — but field removal is a larger struct change). The cleanest option: rename `vx` → `speed` conceptually, zero out `vy`, and document in a comment.

### Pitfall 3: Torus Wrap vs. Respawn Logic Boundary
**What goes wrong:** The existing torus-wrap is 4 lines at the bottom of the starfield animate loop. Replacing it is straightforward, but the respawn must scatter positions uniformly across the entire pad — not just at the entry edge. Stars entering from "behind" (heading reversed) would cluster at one edge if not scattered uniformly.
**Why it happens:** Direction-agnostic respawn (random x, y anywhere) was chosen precisely to avoid this. Do not implement edge-entry respawn.
**Warning signs:** Stars appearing to "pop in" at screen edges when heading is reversed.

### Pitfall 4: juce::Colour::fromHSV Hue Wraparound
**What goes wrong:** `getHue()` returns 0..1 float. Adding ±8° / 360° = ±0.0222 can push hue outside 0..1.
**How to avoid:** Use `std::fmod(baseHue + hueOffset/360.0f + 1.0f, 1.0f)` to ensure wraparound. JUCE's `fromHSV` may clamp rather than wrap.

### Pitfall 5: Shooting Star "Total Distance" Calculation
**What goes wrong:** Shooting star moves at a fixed speed across the pad. The "40–80px long streak" is the visual trail behind the head — not the full crossing distance. The star crosses the pad over ~0.3s total (per CONTEXT.md), so crossing distance = speed × 0.3s.
**How to avoid:** Define `totalDistPx = juce::jmin(w, h) * 0.6f` (crosses ~60% of pad), `durationTicks = 18` (0.3s at 60Hz). Progress advances `1.0f / 18.0f` per tick. streakLen = `juce::jmap(progress, 0.0f, 0.5f, 0.0f, 60.0f)` etc.

### Pitfall 6: Nebula `isRadial` Gradient with Scale Transform
**What goes wrong:** `juce::ColourGradient(isRadial=true)` always draws a circular gradient. To get an ellipse (different rx/ry), you must scale the Graphics context. But if `rx == ry`, the scale is identity and no transform is needed.
**How to avoid:** Always apply the transform even when nearly equal — the cost is negligible and the code path is uniform.

### Drift Heading Update in timerCallback()

```cpp
// Source: CONTEXT.md specification
// Insert BEFORE the starfield animate loop:
{
    const float padW  = (float)getWidth();
    const float padCx = padW * 0.5f;
    const float deflX = (displayCx_ - padCx) / (padW * 0.5f);  // normalized -1..+1
    constexpr float kTurnRate = 0.004f;  // rad/tick at full deflection (~15°/s at 60Hz)
    driftHeading_ += kTurnRate * deflX;
    // No wraparound needed — angle accumulates freely, cos/sin handle it
}
```

### Starfield Animate Loop Replacement in timerCallback()

```cpp
// Source: CONTEXT.md specification
// Replaces existing torus-wrap loop (lines ~1285–1294):
for (auto& s : starfield_)
{
    // Phase 42 large-star freeze (warpRamp_ == 0.0f if Phase 42 not yet implemented)
    if (warpRamp_ > 0.0f && s.r > 1.2f)
        continue;

    const float depthMult = (s.r > 1.4f) ? 1.0f : (s.r > 0.8f) ? 0.6f : 0.3f;
    const float effAngle  = driftHeading_ + s.baseAngle;
    s.x += std::cos(effAngle) * s.speed * depthMult;
    s.y += std::sin(effAngle) * s.speed * depthMult;

    // Respawn with fade-in (replaces torus wrap)
    if (s.x < 0.0f || s.x > 1.0f || s.y < 0.0f || s.y > 1.0f)
    {
        static juce::Random respawnRng(0xC0FFEE43ULL);
        s.x = respawnRng.nextFloat();
        s.y = respawnRng.nextFloat();
        s.twinklePhase = respawnRng.nextFloat() * juce::MathConstants<float>::twoPi;
        s.fadeIn = 30;  // ~0.5s at 60Hz
    }

    // Advance fade-in counter
    if (s.fadeIn > 0)
        --s.fadeIn;
}
```

### Twinkle Phase Advance in timerCallback()

```cpp
// Advance twinkle phase for each star — separate pass after position update
// (or inline with position update above)
for (auto& s : starfield_)
{
    // twinkleFreq stored per-star; range 0.3–0.8 Hz → 0.031–0.084 rad/tick at 60Hz
    // Approximate: store twinkle freq as part of baseAngle sign? No — use a separate approach:
    // Use (s.speed * 3000.0f) as a proxy for per-star frequency variation.
    // Better: store twinkleFreq explicitly, or derive from existing r field.
    // Simplest: s.twinklePhase += (0.031f + s.r * 0.035f)  (r in 0.5–1.6 → freq in 0.031–0.087)
    s.twinklePhase = std::fmod(
        s.twinklePhase + (0.031f + s.r * 0.035f),
        juce::MathConstants<float>::twoPi);
}
```

## Open Questions

1. **`speed` field in `StarDot`**
   - What we know: Current `StarDot { x, y, r, vx, vy, c }` — `vx` and `vy` encode both speed and direction. CONTEXT.md formulas reference `s.speed` as a separate scalar.
   - What's unclear: Does the plan add an explicit `speed` field, or repurpose `vx` as speed scalar (zeroing `vy`)?
   - Recommendation: Add explicit `float speed` to `StarDot`. Keep `vx, vy` zeroed/unused after migration. This avoids a semantic rename that could confuse future readers.

2. **Phase 42 dependency for warpT crossfades**
   - What we know: Phase 42 was reverted. `warpRamp_` does not exist.
   - What's unclear: Will Phase 42 be re-implemented before 43.2 ships?
   - Recommendation: Phase 43.2 Plan 01 adds `float warpRamp_ = 0.0f;` to JoystickPad. All crossfade guards use this. When Phase 42 re-implements `warpRamp_`, it will simply overwrite the field initialization — zero conflicts.

3. **Shooting star: normalized vs. pixel coordinates for position/speed**
   - What we know: Stars use normalized 0..1 coordinates. Shooting star needs pixel-accurate streak length (40–80px).
   - Recommendation: Store shooting star position in normalized coords (like stars), convert to pixels in paint() for streak drawing. Duration in ticks.
