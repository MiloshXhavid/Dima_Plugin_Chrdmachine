## Summary

Phase 24.1 fixes two related UX defects in the left joystick modulation system implemented in Phase 21. The first defect is visual: when the left joystick is routed to an LFO parameter (Freq, Phase, Level) or Gate Length, the audio engine responds correctly but the corresponding UI slider stays frozen. The second defect is behavioral: joystick movement overwrites the slider's base value rather than modulating around it — when the stick springs back to center, the UI shows the displaced value, not the original slider value.

The root cause of both defects is the same: the `writeParam` lambda in `processBlock` uses `raw->store(val, std::memory_order_relaxed)` — a direct atomic write to the APVTS raw value — which bypasses the APVTS listener/notifier chain. `SliderParameterAttachment` and `SliderAtt` listen to parameter change notifications, so they never see this write and the slider thumb does not move. Additionally, because `raw->store` replaces the APVTS value directly, there is no "base + offset" separation: the base value is destroyed on the first joystick movement.

The established pattern in this codebase for cross-thread display updates is the `filterCutDisplay_` / `filterResDisplay_` pattern: the audio thread writes a display atomic, and `timerCallback()` (30 Hz, message thread) reads it and calls `slider.setValue(dontSendNotification)` to update the visual. For center-offset, the audio thread computes `actual = base + joystickOffset` and publishes `actual` to the display atomic. The APVTS always holds the base; the display atomic holds the modulated value. When the stick returns to center, the display atomic converges to the base value on the next 30 Hz tick.

**Confirmed ordering in processBlock:** The LFO params block runs at lines ~692–797 (early), and the filter/dispatch block runs at lines ~1600–1766 (late). This means the dispatch block's computed override values cannot feed into the LFO params block in the same processBlock call. The override takes effect the next block (one-block latency at 256–512 samples — inaudible). The implementation uses private audio-thread-only float members set in dispatch and consumed in the next block's LFO params section.

**Primary recommendation:** Add display atomics (`lfoXRateDisplay_`, `lfoXPhaseDisplay_`, `lfoXLevelDisplay_`, `lfoYRateDisplay_`, `lfoYPhaseDisplay_`, `lfoYLevelDisplay_`, `gateLengthDisplay_`) to `PluginProcessor`. In processBlock's dispatch block, compute `actual = sliderBase + joystickOffset` (clamped) and store to display atomics (replacing `writeParam` calls). Add audio-thread float members for the next-block LFO override. In `timerCallback()`, read display atomics and call `slider.setValue(actual, juce::dontSendNotification)` — updates visual without APVTS feedback loop.

## Architecture Patterns

### processBlock Execution Order (CONFIRMED)

```
processBlock (audio thread):
    Line ~690   buildChordParams()
    Line ~692   LFO params block (reads APVTS lfoXRate, lfoXPhase, lfoXLevel, etc.)
                → lfoX_.process(xp), lfoY_.process(yp)
    Line ~800   chord/trigger/looper/MIDI dispatch
    ...
    Line ~1600  Filter CC + Left Joystick dispatch block (reads filterXMode, prevFilterX_, etc.)
                → writeParam calls (currently; replaced in this phase)
                → stores to display atomics (new)
                → stores to audio-thread float members for NEXT block's LFO override
```

The dispatch block runs AFTER the LFO block. Override values set in the dispatch block are consumed by the LFO block in the **next** processBlock call. One-block latency (~6ms at 256 samples / 44.1kHz) — inaudible.

### Current Flow (Broken)

```
processBlock dispatch block:
    writeParam("lfoXRate", hz)
        → apvts.getRawParameterValue("lfoXRate")->store(hz)   // bypasses listener chain
        → SliderParameterAttachment does NOT see this write
        → lfoXRateSlider_ thumb does NOT move
        → APVTS base value is permanently overwritten with joystick-driven hz
        → when stick returns to center, "base" is now the last joystick position
```

### Target Flow (Center-Offset + Visual Tracking)

```
processBlock LFO block (reads prev-block override if active, else APVTS):
    xp.rateHz = lfoXRateOverride_  // set by dispatch last block; fallback = APVTS value

processBlock dispatch block:
    base   = apvts.getRawParameterValue("lfoXRate")->load()   // APVTS never written
    offset = stickVal * atten * 9.995f                        // signed offset
    actual = jlimit(0.01f, 20.0f, base + offset)
    lfoXRateDisplay_.store(actual)                            // publish for UI
    lfoXRateOverride_ = actual                                // for next block's LFO

timerCallback() 30 Hz (message thread):
    lfoXRateSlider_.setValue(lfoXRateDisplay_.load(), dontSendNotification)
    // Thumb follows stick in real time; no APVTS write; no feedback loop
    // When stick releases: offset → 0, actual → base, slider snaps back
```

### Pattern: Display Atomic + Timer Poll

This is identical to the existing `filterCutDisplay_` / `filterResDisplay_` pattern.

**What:** Audio thread publishes computed display value to `std::atomic<float>`. Message thread reads it at 30 Hz in `timerCallback()` and updates slider visual via `setValue(dontSendNotification)`.

**Why `dontSendNotification`:** `setValue(dontSendNotification)` moves the thumb without firing the APVTS attachment's `parameterValueChanged` callback, avoiding a feedback loop. The APVTS value stays at the user's base value permanently.

**Key design invariant:** APVTS holds the base. Display atomic holds the modulated value. Override float holds the "carry-forward" for the next block's LFO params. All three are independent — no circular dependency.

### Center-Offset Mapping for Each Target

| Mode | APVTS Base Param | Offset Magnitude | Clamped Range | Display Atomic | Override Member |
|------|-----------------|-----------------|---------------|----------------|-----------------|
| LFO-X Freq (free) | lfoXRate (0.01–20 Hz) | stickVal * atten * 9.995f | 0.01–20 Hz | lfoXRateDisplay_ | lfoXRateOverride_ |
| LFO-X Freq (sync) | N/A — lfoXSubdivMult_ is already center-relative (mult=1.0 at center) | unchanged | unchanged | N/A | N/A |
| LFO-X Phase | lfoXPhase (0–360°) | stickVal * atten * 180.0f | 0–360° | lfoXPhaseDisplay_ | lfoXPhaseOverride_ |
| LFO-X Level | lfoXLevel (0–1) | stickVal * atten * 0.5f | 0–1 | lfoXLevelDisplay_ | lfoXLevelOverride_ |
| LFO-Y Freq (free) | lfoYRate (0.01–20 Hz) | stickVal * atten * 9.995f | 0.01–20 Hz | lfoYRateDisplay_ | lfoYRateOverride_ |
| LFO-Y Phase | lfoYPhase (0–360°) | stickVal * atten * 180.0f | 0–360° | lfoYPhaseDisplay_ | lfoYPhaseOverride_ |
| LFO-Y Level | lfoYLevel (0–1) | stickVal * atten * 0.5f | 0–1 | lfoYLevelDisplay_ | lfoYLevelOverride_ |
| Gate Length | gateLength (0–1) | stickVal * atten * 0.5f | 0–1 | gateLengthDisplay_ | N/A (read by SmoothedValue target) |

> Note: The Phase 21 mapping for free-mode Freq used `norm = (stickVal * atten + 1.0) * 0.5; hz = 0.01 + norm * (20.0 - 0.01)`. This mapped stick center to ~10 Hz (mid-range), ignoring the slider base. The new mapping uses `sliderBase + offset` so stick center always returns to whatever the user set on the slider.

### LFO Override Members (PluginProcessor private, audio-thread only)

```cpp
// Default to APVTS value sentinel — overridden each block when joystick is active.
// Reset to -1.0f when xMode/yMode changes so LFO block falls back to APVTS.
float lfoXRateOverride_  = -1.0f;   // -1 = inactive; use APVTS lfoXRate
float lfoXPhaseOverride_ = -1.0f;
float lfoXLevelOverride_ = -1.0f;
float lfoYRateOverride_  = -1.0f;
float lfoYPhaseOverride_ = -1.0f;
float lfoYLevelOverride_ = -1.0f;
```

These are plain floats (not atomics) — they are only ever accessed on the audio thread. No synchronization needed.

### LFO Block Consumption of Override

```cpp
// In LFO params block (lines ~715, ~727, ~729 area):
xp.rateHz     = (lfoXRateOverride_  >= 0.0f) ? lfoXRateOverride_
                                               : *apvts.getRawParameterValue(ParamID::lfoXRate);
xp.phaseShift = ((lfoXPhaseOverride_ >= 0.0f) ? lfoXPhaseOverride_
                                               : *apvts.getRawParameterValue(ParamID::lfoXPhase)) / 360.0f;

                                              : *apvts.getRawParameterValue(ParamID::lfoXLevel);
```

Reset overrides to -1.0f when mode is not active (in the dispatch block's else/default path), so when the dropdown is switched back to CC mode, the LFO resumes reading APVTS directly.

### Gate Length Smoothing (Third TODO)

**Root cause:** `gateLength` is read raw each block with no smoothing. Rapid joystick movements produce stepped note-off timing that sounds zipper-noisy.

**Fix:** `juce::SmoothedValue<float, juce::ValueSmoothingTypes::Linear>` in PluginProcessor (private). Prepare in `prepareToPlay`. Each processBlock, set target and advance. The target source changes based on whether gate is joystick-driven.

**`SmoothedValue` is in `juce_core`** — no `juce_dsp` needed (juce_dsp excluded per REQUIREMENTS.md Out of Scope section).

### Anti-Patterns to Avoid

- **Writing joystick-modulated values into APVTS from audio thread:** `setValueNotifyingHost` is illegal from audio thread (crosses to message thread). `raw->store` avoids the race but destroys the base value. Use display atomics and override floats instead.
- **Calling `slider.setValue` from audio thread:** Illegal — JUCE components are not thread-safe. All slider calls must be from the message thread (timerCallback).
- **Using `setValueNotifyingHost` in timerCallback for display updates:** Triggers attachment listener → `setValue(sendNotification)` → feedback loop. Use `slider.setValue(val, dontSendNotification)` directly.
- **SmoothedValue without `reset()` in `prepareToPlay`:** SmoothedValue internal buffer size is derived from sample rate and ramp time. Must call `reset(sampleRate, rampSeconds)` before any `skip()` or `getNextValue()` calls.
- **Forgetting to reset override floats to -1.0f when mode changes:** If the user switches xMode from LFO-X Freq to CC74, the override float will retain the last joystick-driven Hz value, poisoning the LFO frequency for the next block. Reset to -1.0f at the top of the dispatch block (before the switch) and only set if mode matches.

## Common Pitfalls

### Pitfall 1: Feedback Loop via setValueNotifyingHost in timerCallback

**What goes wrong:** `pParam->setValueNotifyingHost(actual)` in timerCallback → APVTS notifies SliderAttachment → attachment calls `slider.setValue(sendNotification)` → onValueChange fires → potential loop or double-update.
**Why it happens:** `setValueNotifyingHost` notifies all listeners including the SliderParameterAttachment.
**How to avoid:** Call `slider.setValue(displayAtomic.load(), dontSendNotification)` directly on the slider widget. The APVTS base value is never modified.
**Warning signs:** Slider thumb oscillates or snaps when joystick moves.

### Pitfall 2: writeParam Polluting the Base Value (current bug)

**What goes wrong:** `raw->store(hz)` permanently replaces the APVTS value. When stick returns to center, the "slider base" is now the last joystick position.
**Why it happens:** APVTS raw float serves as both base store and modulated value — no separation.
**How to avoid:** Never write joystick-modulated values into APVTS. Compute `actual = base + offset` in dispatch, pass to override float for next block's LFO, publish to display atomic. APVTS stays at base.
**Warning signs:** After releasing joystick, slider does not return to where the user set it.

### Pitfall 3: LFO Playback Mode Conflict

**What goes wrong:** When LFO recording is in Playback state, `lfoXPhaseSlider_` / `lfoXLevelSlider_` / `lfoXRateSlider_` are `setEnabled(false)` and `setAlpha(0.35f)`. A timerCallback that also calls `slider.setValue()` will visually move a grayed-out slider.
**Why it happens:** The display-update block and the LFO-playback-dim block both touch the same sliders.
**How to avoid:** Guard display-update `slider.setValue()` calls with `if (!xPlayback)` / `if (!yPlayback)` — same booleans already computed in the existing LFO recording UI block in timerCallback.
**Warning signs:** Grayed-out sliders move during LFO playback mode.

### Pitfall 4: Override Float Not Reset on Mode Change

**What goes wrong:** User sets xMode=2 (LFO-X Freq), moves joystick, lfoXRateOverride_ = 18.5 Hz. User then changes xMode to 0 (CC74). Next processBlock: LFO block reads lfoXRateOverride_=18.5 and uses it even though LFO-X Freq is no longer the target — LFO runs at wrong rate.
**Why it happens:** Override float is only set in the dispatch when xMode==2 but not cleared when xMode changes.
**How to avoid:** At the top of the dispatch block (before the switch), reset all override floats to -1.0f unconditionally. The switch then sets them only for the active mode. If mode is not 2/3/4, the override stays -1.0f and the LFO block falls back to APVTS.
**Warning signs:** LFO rate/phase/level "sticks" at wrong value after changing dropdown mode.

### Pitfall 5: SmoothedValue Skip vs. getNextValue

**What goes wrong:** Using `getNextValue()` once per block instead of per sample, or skipping `skip(numSamples)` entirely.
**Why it happens:** SmoothedValue is per-sample; `skip(N)` advances N steps efficiently for block-level use.
**How to avoid:** Use `skip(buffer.getNumSamples())` then `getCurrentValue()`. Gate timing is block-level (not sample-level), so this is accurate enough.
**Warning signs:** Smoothing ramp is much slower or faster than expected.

### Pitfall 6: Phase Slider Range Clamp at 0/360

**What goes wrong:** `sliderBase + offset` exceeds 360° or goes below 0° — slider hits a wall at the extreme.
**Why it happens:** APVTS lfoXPhase is defined as 0–360 (hard clamped). Phase wraps mathematically but APVTS does not wrap.
**How to avoid:** `juce::jlimit(0.0f, 360.0f, base + offset)` for both display atomic and LFO override. Clamping is simpler than modular wrap and consistent with existing APVTS limits. Users rarely need the extreme 0/360 while also holding full joystick deflection.
**Warning signs:** Phase slider gets stuck at 0 or 360 during performance.

### Pattern 3: Dispatch Block — Replace writeParam with Center-Offset

```cpp
// In processBlock, LFO/Gate dispatch block — replaces writeParam calls:
// Source: refactoring of existing block at PluginProcessor.cpp lines 1681–1765

// Reset all overrides before dispatch — cleared if mode does not match
lfoXRateOverride_ = lfoXPhaseOverride_ = lfoXLevelOverride_ = -1.0f;
lfoYRateOverride_ = lfoYPhaseOverride_ = lfoYLevelOverride_ = -1.0f;

if (stickUpdated)
{
    if (xMode >= 2 && xMode <= 5)
    {
        const float stick = prevFilterX_ * (xAtten / 100.0f);  // -1..+1 attenuated

        switch (xMode)
        {
            case 2:  // LFO-X Freq
            {
                const bool syncOn = *apvts.getRawParameterValue(ParamID::lfoXSync) > 0.5f;
                if (syncOn)
                {
                    // Sync mode: existing mult approach is already center-relative
                    lfoXSubdivMult_.store(std::pow(4.0f, stick), std::memory_order_relaxed);
                }
                else
                {
                    const float base   = apvts.getRawParameterValue(ParamID::lfoXRate)->load();
                    const float actual = juce::jlimit(0.01f, 20.0f, base + stick * 9.995f);
                    lfoXRateDisplay_.store(actual, std::memory_order_relaxed);
                    lfoXRateOverride_ = actual;  // consumed by LFO block next processBlock
                }
                break;
            }
            case 3:  // LFO-X Phase
            {
                const float base   = apvts.getRawParameterValue(ParamID::lfoXPhase)->load();
                const float actual = juce::jlimit(0.0f, 360.0f, base + stick * 180.0f);
                lfoXPhaseDisplay_.store(actual, std::memory_order_relaxed);
                lfoXPhaseOverride_ = actual;
                break;
            }
            case 4:  // LFO-X Level
            {
                const float base   = apvts.getRawParameterValue(ParamID::lfoXLevel)->load();
                const float actual = juce::jlimit(0.0f, 1.0f, base + stick * 0.5f);
                lfoXLevelDisplay_.store(actual, std::memory_order_relaxed);
                lfoXLevelOverride_ = actual;
                break;
            }
            case 5:  // Gate Length (no LFO override needed — smoothed gate reads display atomic)
            {
                const float base   = apvts.getRawParameterValue(ParamID::gateLength)->load();
                const float actual = juce::jlimit(0.0f, 1.0f, base + stick * 0.5f);
                gateLengthDisplay_.store(actual, std::memory_order_relaxed);
                break;
            }
        }
    }
    // ... same pattern for Y axis ...
}
```

## Open Questions

1. **LFO Freq Rate slider in Sync mode** — RESOLVED (no action needed)
   - What we know: When syncOn, the rate slider shows the subdivision choice (0–5 index). The joystick-to-freq-sync target uses `lfoXSubdivMult_` which has no slider. The mult is already center-relative (std::pow(4,0)=1.0 at center). No display atomic or slider update needed for sync mode.
   - Resolution: Leave rate slider static during sync mode. Center-offset is already implemented via the multiplicative approach.

2. **Gate Length display when both X and Y target gate simultaneously** — RESOLVED (document as limitation)
   - What we know: If xMode==5 and yMode==5 both target gateLength, both dispatch blocks write to `gateLengthDisplay_`. The Y dispatch runs after X, so Y's value wins.
   - Resolution: This matches existing `writeParam` behavior. Document as known limitation. Recommend the dropdown UI prevent both axes from targeting gate simultaneously (not in scope for this phase).

3. **One-block LFO latency** — RESOLVED (acceptable)
