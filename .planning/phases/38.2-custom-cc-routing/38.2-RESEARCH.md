- When selected, the combo **shrinks** to make room for an **editable text field** inline in the same row

## Summary

Phase 38.2 adds "Custom CC..." as a final item in four ComboBoxes so users can route to any MIDI CC 0–127. The design is: user picks "Custom CC...", an inline `juce::Label` (setEditable) appears in the same row replacing part of the combo's width, user types and presses Enter. The chosen number is stored in a new APVTS float param per combo and the combo's displayed text updates to `"CC [n]"`.

The codebase already has exactly this pattern for the BPM display (`bpmDisplayLabel_.setEditable(false, true)` with `onTextChange` lambda, in PluginEditor.cpp around line 2645–2655). That pattern is the template to replicate.

The processor side is also clean: each of the four combos has a distinct lookup path. `lfoXCcDestBox_` and `lfoYCcDestBox_` use the `ccDestToNumber()` helper (static lookup table at line 607). `filterXModeBox_` and `filterYModeBox_` use `kFilterCcNums[]` array with `mode - kNumLfoTargetModes` offset (line 2024–2025). Both paths need a "custom CC" branch that reads the corresponding APVTS float param instead of the lookup table.

**Primary recommendation:** Use `juce::Label::setEditable(false, true)` for inline text entry — it is already proven in this codebase, requires no new JUCE components, and uses `onTextChange` for confirmation.

## Architecture Patterns

### Combo + Inline Label Pattern (BPM template)

The existing BPM editable label in PluginEditor.cpp shows the exact pattern:

```cpp
// Source: PluginEditor.cpp ~line 2645
bpmDisplayLabel_.setColour(juce::Label::backgroundWhenEditingColourId, Clr::panel.darker(0.3f));
bpmDisplayLabel_.setColour(juce::Label::textWhenEditingColourId, Clr::text);
bpmDisplayLabel_.setEditable(false, true);  // click-once (not double-click) to edit
bpmDisplayLabel_.onTextChange = [this]() {
    const float val = bpmDisplayLabel_.getText()
        .retainCharacters("0123456789.").getFloatValue();
    if (val >= 30.0f && val <= 240.0f) { /* write to APVTS */ }
};
```

For custom CC entry, adapt to `(false, true)` so single-click starts editing (consistent with BPM). The `onTextChange` fires when the user commits (presses Enter or clicks away). Clamp silently to 0–127.

### Where "Custom CC..." Lives in Each Combo's Item List

**lfoXCcDestBox_ and lfoYCcDestBox_** — currently 19 items (IDs 1–19):
- Item 1 = "Off" (index 0)
- Items 2–19 = named CCs (indices 1–18, maps via `ccDestToNumber()`)
- **New item 20 = "Custom CC..."** (index 19) — appended last, no existing index shifts

**filterXModeBox_ and filterYModeBox_** — currently 25 items (IDs 1–25):
- Items 1–7 = LFO/Gate targets (indices 0–6)
- Items 8–25 = 18 named CCs (indices 7–24, maps via `kFilterCcNums[mode - 7]`)
- **New item 26 = "Custom CC..."** (index 25) — appended last, no existing index shifts

Existing APVTS `AudioParameterChoice` layouts map directly to 0-based choice indices. Both `filterXMode` and `filterYMode` are declared with `addChoice()` — the StringArray length must match. Adding one entry to the StringArray and one item to the ComboBox must happen in both `createParameterLayout()` (PluginProcessor.cpp) and the editor combo population (PluginEditor.cpp) together. Similarly, `lfoXCcDest` and `lfoYCcDest` are `AudioParameterChoice` — their StringArray in `createParameterLayout` must gain "Custom CC" as last entry.

### APVTS Parameter Layout for Custom CC Storage

Use `AudioParameterFloat` with a step of 1.0 and range 0–127:

```cpp
// In createParameterLayout(), PluginProcessor.cpp
layout.add(std::make_unique<juce::AudioParameterFloat>(
    "lfoXCustomCc", "LFO X Custom CC",
    juce::NormalisableRange<float>(0.0f, 127.0f, 1.0f), 74.0f));  // default: CC74
layout.add(std::make_unique<juce::AudioParameterFloat>(
    "lfoYCustomCc", "LFO Y Custom CC",
    juce::NormalisableRange<float>(0.0f, 127.0f, 1.0f), 71.0f));  // default: CC71
layout.add(std::make_unique<juce::AudioParameterFloat>(
    "filterXCustomCc", "Filter X Custom CC",
    juce::NormalisableRange<float>(0.0f, 127.0f, 1.0f), 74.0f));
layout.add(std::make_unique<juce::AudioParameterFloat>(
    "filterYCustomCc", "Filter Y Custom CC",
    juce::NormalisableRange<float>(0.0f, 127.0f, 1.0f), 71.0f));
```

The defaults should match the last named CC the user was most likely using — defaults matching existing filterXMode (CC71) and filterYMode (CC74) are appropriate to avoid unexpected jumps on first use.

### Processor-Side CC Dispatch Changes

**ccDestToNumber() (line 607)** — currently a static array lookup:
```cpp
static int ccDestToNumber(int idx)
{
    static const int t[] = {-1,1,2,5,7,10,11,12,16,17,71,72,73,74,75,76,77,91,93};
    return (idx >= 0 && idx < (int)std::size(t)) ? t[idx] : -1;
}
```
This handles indices 0–18 (19 items). When "Custom CC..." is item index 19, this function must gain a branch: if `idx == 19`, read `apvts.getRawParameterValue("lfoXCustomCc")->load()` (or Y variant). However, `ccDestToNumber()` is a free function with no access to `apvts`. Options:
- Pass the custom CC value as a second parameter: `ccDestToNumber(int idx, int customCcVal)` — clean, no shared state
- Or inline the lookup at the two call sites (lines 2560–2561) — also acceptable given it is only 2 callers

**filterXMode/filterYMode dispatch (lines 2021–2025)**:
```cpp
const int xMode  = (int)apvts.getRawParameterValue("filterXMode")->load();
const int yMode  = (int)apvts.getRawParameterValue("filterYMode")->load();
const int ccXnum = (xMode >= kNumLfoTargetModes) ? kFilterCcNums[xMode - kNumLfoTargetModes] : -1;
const int ccYnum = (yMode >= kNumLfoTargetModes) ? kFilterCcNums[yMode - kNumLfoTargetModes] : -1;
```
When index is 25 (= 7 + 18, the new "Custom CC..." slot), `xMode - kNumLfoTargetModes` = 18, which is out of bounds for `kFilterCcNums[18]` (array has indices 0–17). A guard is required:
```cpp
// kNumFilterCcModes = 18 (named CCs), kNumLfoTargetModes = 7
// New "Custom CC..." is at index 7+18 = 25
const bool xIsCustom = (xMode == kNumLfoTargetModes + kNumFilterCcModes);
const int ccXnum = xIsCustom
    ? (int)apvts.getRawParameterValue("filterXCustomCc")->load()
    : (xMode >= kNumLfoTargetModes) ? kFilterCcNums[xMode - kNumLfoTargetModes] : -1;
```
Same pattern for Y. This pattern appears at multiple locations in processBlock where filterXMode/filterYMode are consumed (lines 2021, 2565, and the LFO center-of-oscillation lookup at lines 2567–2568 — all these sites must be updated consistently).

### UI Show/Hide + Dynamic Label Pattern

When "Custom CC..." is selected, the combo shrinks to about 60% of its row width and a `juce::Label` appears beside it in the remaining space. The `onChange` lambda handles this:

```cpp
lfoXCcDestBox_.onChange = [this]() {
    const bool customActive = (lfoXCcDestBox_.getSelectedId() == 20); // "Custom CC..." item
    lfoXCustomCcLabel_.setVisible(customActive);
    if (customActive) {
        // show last known CC number as starting value
        const int lastCc = (int)*proc_.apvts.getRawParameterValue("lfoXCustomCc");
        lfoXCustomCcLabel_.setText(juce::String(lastCc), juce::dontSendNotification);
    } else {
        // Update combo text to "CC [n]" when returning from custom...
        // handled in timerCallback or directly here
    }
    resized(); // trigger layout recalculation
};
```

The `resized()` block checks `lfoXCustomCcLabel_.isVisible()` to decide whether to split the row. The combo shows `"CC [n]"` while custom is active — set via `lfoXCcDestBox_.setText("CC " + juce::String(n))` (note: this bypasses APVTS attachment display, which is fine since the attachment stores an index).

**Important:** `juce::ComboBox::setText()` sets the displayed text without changing the selected item index. This is the correct mechanism to show "CC 42" while the item "Custom CC..." remains selected (selectedId = 20).

### Text Confirmed → Write to APVTS

```cpp
lfoXCustomCcLabel_.onTextChange = [this]() {
    const int val = juce::jlimit(0, 127,
        lfoXCustomCcLabel_.getText().getIntValue());
    if (auto* p = proc_.apvts.getParameter("lfoXCustomCc"))
        p->setValueNotifyingHost(p->getNormalisableRange().convertTo0to1((float)val));
    // Update combo display text
    lfoXCcDestBox_.setText("CC " + juce::String(val), juce::dontSendNotification);
};
```

### timerCallback Update for "CC [n]" display

In `timerCallback()`, if the custom CC is active and the stored APVTS value has changed (e.g. from automation or preset load), refresh the combo text label to keep it in sync:

```cpp
// In timerCallback():
if (lfoXCcDestBox_.getSelectedId() == 20) {
    const int n = (int)*proc_.apvts.getRawParameterValue("lfoXCustomCc");
    lfoXCcDestBox_.setText("CC " + juce::String(n), juce::dontSendNotification);
}
```

This costs nothing (30 Hz, no repaint unless string changed).

### Preset Save / Load Behavior

APVTS handles all four new float params automatically via `getStateInformation` / `setStateInformation`. The `AudioParameterChoice` for each combo also saves correctly — on load, if the choice index is the "Custom CC..." slot, the combo will show that item and the `onChange` lambda will fire, making the label visible. The APVTS float param for the custom value will also be restored. No special serialization logic is needed.

### Backward Compatibility

Presets saved before this phase will NOT have the four new float params. JUCE inserts missing params at their default value on load — no crash. The `AudioParameterChoice` StringArrays for `lfoXCcDest`, `lfoYCcDest`, `filterXMode`, `filterYMode` are now longer by one entry. Old presets store a choice index — those indices are still valid (they still map to the same items). The new "Custom CC..." item is at the end, so no shift.

## Common Pitfalls

### Pitfall 1: AudioParameterChoice StringArray Length Mismatch
**What goes wrong:** APVTS `AudioParameterChoice` for `filterXMode` has a StringArray declared in `createParameterLayout()`. The ComboBox in the editor also populates items. If the StringArray in the layout gains a "Custom CC" entry but the ComboBox doesn't (or vice versa), APVTS will store an out-of-range choice index and the parameter will clamp or behave unexpectedly.
**How to avoid:** Add "Custom CC" to both the layout StringArray and the editor `addItem()` call in the same wave.
**Warning signs:** Combo shows wrong item on preset reload; APVTS choice index one higher than the number of combo items.

### Pitfall 2: kFilterCcNums[] Out-of-Bounds Access
**What goes wrong:** `kFilterCcNums[xMode - kNumLfoTargetModes]` is called where `xMode` is now 25 (the new custom slot). `kNumLfoTargetModes = 7`, so the index becomes 18, but `kFilterCcNums` has exactly 18 elements (indices 0–17). Array access at index 18 is undefined behavior.
**How to avoid:** Add a bounds check before the array access at every site in processBlock where filterXMode/filterYMode is consumed. There are at minimum 4 such sites (lines ~2024, ~2567, ~2568, and the LFO center-offset calculation). All must be guarded.
**Warning signs:** Occasional CC spikes or wrong CC numbers sent when custom mode is active.

### Pitfall 3: ccDestToNumber() Receives Index 19 (custom)
**What goes wrong:** `ccDestToNumber(19)` returns -1 (no CC) because the table only has 19 entries (indices 0–18). When the user has selected custom, the LFO CC will silently be disabled.
**How to avoid:** Extend the function or add a callsite check. The cleanest approach: pass the resolved custom CC value as a parameter so the function never needs apvts access.
**Warning signs:** LFO CC output stops when custom is selected.

### Pitfall 4: ComboBox setText() Being Overridden by APVTS Attachment
**What goes wrong:** `lfoXCcDestAtt_` is a `ComboBoxAttachment`. When APVTS notifies the attachment of a value change (e.g. during preset load), the attachment calls `comboBox.setSelectedId(...)` which resets the displayed text from the item list, overriding any `setText("CC 42")` call.
**How to avoid:** After the attachment writes to the combo, also call `setText("CC [n]")` again in the `onChange` callback or in timerCallback. The approach in the Architecture Patterns section (timerCallback polling) handles this correctly.
**Warning signs:** "CC 42" reverts to "Custom CC..." text after preset load.

### Pitfall 5: Label Visibility Not Updated in resized() After Preset Load
**What goes wrong:** On preset load, APVTS restores `filterXMode` to the "Custom CC..." index, but `resized()` is only called during initial layout — not when APVTS restores values. The label may remain hidden.
**How to avoid:** In `timerCallback()`, detect when the combo's selected ID matches the "Custom CC..." slot and ensure the label visibility is set correctly. A one-time flag pattern (similar to `lastHighlightMode_` already in PluginEditor) can avoid doing this every 30 Hz tick.
**Warning signs:** Custom CC label invisible after preset reload; combo shows "Custom CC..." but no text field appears.

### Pitfall 6: SWAP INV Swaps Filter Combo Attachments
**What goes wrong:** Lines 4753–4761 in PluginEditor.cpp swap the APVTS attachments of `filterXModeBox_` and `filterYModeBox_` when the SWAP button is toggled. After this phase, if "Custom CC..." is active on one of those combos, the swap will correctly swap the attachment but the custom CC label position and the `onChange` lambda closure (capturing the specific label) will not reflect the swap.
**How to avoid:** The SWAP feature touches only attachments, not the combos themselves (which remain in fixed screen positions). The custom CC labels are per-combo-widget, not per-APVTS-param. After a SWAP, `filterXModeBox_`'s text field stores `filterYCustomCc` (because the attachment now points to `filterYMode`) but the label writes to `filterXCustomCc`. This mismatch is a subtle bug.

**Resolution options:**
1. On SWAP toggle, also swap which `customCcLabel_` maps to which APVTS param — mirror the attachment swap pattern exactly.
2. Or: read the correct custom CC param by examining which attachment is currently connected.

Option 1 is cleaner. The swap block at line 4751 must be extended to also swap the `onTextChange` lambdas (or the APVTS param ID strings) of the two filter custom CC labels.

## Open Questions

1. **SWAP button interaction with custom CC labels**
   - What we know: SWAP (line 4751) swaps only APVTS attachments for `filterXModeBox_`/`filterYModeBox_`, not their positions or `onChange` lambdas.
   - What's unclear: The cleanest implementation of swapping which custom CC param a given label writes to.
   - Recommendation: Planner should mirror the attachment swap to also swap the label-to-param binding. This can be done by reassigning the `onTextChange` lambda on each SWAP toggle, exactly as `filterXModeAtt_` is reassigned.

2. **Initial combo text on plugin load when custom is saved**
   - What we know: On load, APVTS restores the choice index. The combo's `onChange` fires. But `resized()` must also be called to lay out the label.
   - Recommendation: The `onChange` lambda should call `resized()` (the editor already has no issues calling this at runtime — it is called whenever the window is resized). This ensures layout is correct regardless of load order.

3. **filterXModeBox / filterYModeBox have 25 items currently vs 26 needed**
   - The `AudioParameterChoice` StringArray in `createParameterLayout()` at line ~233 must gain one entry to match. Verify that the existing `addChoice("filterXMode", ..., xModes, 16)` default index (16) remains correct — adding one item at the end does not shift existing indices.
   - Confirmation: adding at end is safe, default index 16 still means "CC71 Resonance".
