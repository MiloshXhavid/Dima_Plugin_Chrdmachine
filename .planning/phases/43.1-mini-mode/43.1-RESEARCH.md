## User Constraints (from CONTEXT.md)

### Locked Decisions

**Window Mode States:** Three states — Full (normal 1120×840×scale), Mini (pad square at current scale, ~308×scale), Maxi (pad square filling available display). Cycled via two dedicated header bar buttons. Overlay button (top-right corner of pad, in Mini/Maxi) returns to Full.

**Mini size:** `padSize = roundToInt(308 * scaleFactor_)`. At 0.75x → 231px, at 1.0x → 308px.

**Maxi size:** `juce::Desktop::getInstance().getDisplays().getDisplayContaining(getScreenBounds().getCentre()).userArea` → `min(width, height)` square. Phase 43 scale factor ignored in Maxi.

**Content in Mini/Maxi:** JoystickPad only — starfield, warp, spring cursor, crosshair, burst particles. All other components hidden. In Maxi: `joystickPad_.setBounds(0, 0, windowSize, windowSize)`.

**Persistence:** Window mode NOT saved across sessions. Always opens Full.

**Implementation approach:**
- `enum class WindowMode { Full, Mini, Maxi }` + `WindowMode windowMode_` in PluginEditor
- `applyWindowMode()` function: hides/shows components, calls `setSize`
- Overlay button: drawn in `paintOverChildren` when `windowMode_ != Full` — 24×24px, top-right, 4px inset
- Mouse hit for overlay: checked in `mouseDown` BEFORE forwarding to children
- Header buttons: two 20×20px buttons at far-right of header strip — ◱ (Mini), ⛶ (Maxi)
- Paint guards: divider lines, logo, footer text guarded with `if (windowMode_ == Full)`

**Gamepad Option Mode extension:** 3 → 5 states (`% 3` → `% 5`). Modes 3=MINI (teal), 4=MAXI (amber). `timerCallback` detects mode transitions into/out of 3/4 and calls `applyWindowMode()`. D-pad dispatch: change `!= 0` to `== 1 || == 2` so modes 3/4 use mode-0 looper behavior. Overlay return resets `optionMode_` to 0.

## Summary

Phase 43.1 adds a 3-state window mode toggle to the plugin editor: Full (current normal view), Mini (pad-only small window at current scale), and Maxi (pad-only fills the display as a square). The implementation is purely a PluginEditor concern — no processor changes needed. The core challenge is managing the transition cleanly: `setSize` on a JUCE AudioProcessorEditor triggers `resized()`, and `resized()` currently assumes Full layout. The solution is to gate all layout work in `resized()` behind a `windowMode_` check, allowing the Full-mode layout to run only when `windowMode_ == Full`, and the Mini/Maxi paths to simply set `joystickPad_.setBounds(0, 0, w, h)`.

The second challenge is the `setResizeLimits` / `setFixedAspectRatio` constraint chain established by Phase 43. These constraints must be suspended or replaced when entering Mini/Maxi mode (since a square window violates the 4:3 aspect ratio constraint), and restored on return to Full. This is the most technically involved part of the phase.

The gamepad optionMode_ cycle extension is a mechanical change in GamepadInput.cpp (one integer mod value) plus timerCallback dispatch routing — straightforward but must be coordinated with the D-pad dispatch condition change.

**Primary recommendation:** Implement `applyWindowMode()` as the single entry point for all mode transitions. It disables the aspect ratio constraint before calling `setSize` in Mini/Maxi, then re-enables 4:3 on return to Full.

## Architecture Patterns

### Pattern 1: Window Mode State Machine

The `windowMode_` member drives all layout decisions. A single `applyWindowMode(WindowMode)` function is the only place that calls `setSize` and changes component visibility.

```cpp
// In PluginEditor.h
enum class WindowMode { Full, Mini, Maxi };
WindowMode windowMode_ { WindowMode::Full };

// In PluginEditor.cpp — applyWindowMode
void PluginEditor::applyWindowMode(WindowMode newMode)
{
    if (windowMode_ == newMode) return;
    windowMode_ = newMode;

    const bool isFull = (newMode == WindowMode::Full);

    // 1. Show/hide all non-pad components
    setAllNonPadComponentsVisible(isFull);

    // 2. Compute target size
    int targetW, targetH;
    if (newMode == WindowMode::Full)
    {
        targetW = juce::roundToInt(1120.0f * scaleFactor_);
        targetH = juce::roundToInt(840.0f  * scaleFactor_);
    }
    else if (newMode == WindowMode::Mini)
    {
        const int padSize = juce::roundToInt(308.0f * scaleFactor_);
        targetW = padSize;
        targetH = padSize;
    }
    else  // Maxi
    {
        const auto display = juce::Desktop::getInstance()
            .getDisplays().getDisplayContaining(getScreenBounds().getCentre());
        const int sq = juce::jmin(display.userArea.getWidth(),
                                   display.userArea.getHeight());
        targetW = sq;
        targetH = sq;
    }

    // 3. Adjust constraints BEFORE setSize
    if (isFull)
    {
        setResizeLimits(840, 630, 1120, 840);
        getConstrainer()->setFixedAspectRatio(1120.0 / 840.0);
    }
    else
    {
        // Square mode — disable aspect ratio lock, set 1:1 limits
        getConstrainer()->setFixedAspectRatio(0.0);  // 0 = unconstrained
        setResizeLimits(targetW, targetH, targetW, targetH);  // exact size only
    }

    // 4. Resize
    setSize(targetW, targetH);
    // setSize triggers resized(), which re-routes to mini/maxi layout
}
```

### Pattern 2: resized() Mode Dispatch

The existing `resized()` becomes a dispatcher. All current layout code runs only in the Full branch.

```cpp
void PluginEditor::resized()
{
    scaleFactor_ = (float)getWidth() / 1120.0f;
    pixelLaf_.setScaleFactor(scaleFactor_);
    proc_.savedUiScale_.store((double)scaleFactor_);

    if (windowMode_ == WindowMode::Full)
    {
        // ... all existing layout code unchanged ...
    }
    else  // Mini or Maxi — pad-only layout
    {
        joystickPad_.setBounds(0, 0, getWidth(), getHeight());
        joystickPad_.setVisible(true);
        // All other components already hidden by applyWindowMode
    }
}
```

**Critical:** `scaleFactor_` derivation and `savedUiScale_.store()` run unconditionally at the top of `resized()` — this is correct for Full mode (host-drag resize). In Mini/Maxi the scale doesn't matter for layout but the store does no harm.

**However:** In Mini mode, `getWidth() / 1120.0f` produces a small scale value (e.g. 0.27 at 308px wide). The `proc_.savedUiScale_.store()` call must be guarded by `if (windowMode_ == WindowMode::Full)` to avoid persisting the Mini-mode "scale" as the saved scale factor.

### Pattern 3: Overlay Button — paintOverChildren + mouseDown

The overlay button is not a `juce::TextButton` — it is drawn directly in `paintOverChildren` and hit-tested in `mouseDown`. This pattern avoids adding a child component that would conflict with the "hide all" pass.

```cpp
// In PluginEditor — compute overlay button rect
juce::Rectangle<int> getOverlayButtonBounds() const
{
    const int sz = 24;
    const int inset = 4;
    return { getWidth() - sz - inset, inset, sz, sz };
}

// paintOverChildren
void PluginEditor::paintOverChildren(juce::Graphics& g)
{
    // ... existing beat pulse dot ...

    if (windowMode_ != WindowMode::Full)
    {
        const auto r = getOverlayButtonBounds();
        g.setColour(juce::Colours::black.withAlpha(0.55f));
        g.fillRect(r);
        g.setColour(juce::Colours::white.withAlpha(0.80f));
        g.setFont(pixelFont_.withHeight(14.0f));
        g.drawText(juce::String::charToString(0x21A9), r,  // ↩ unicode
                   juce::Justification::centred);
    }
}

// mouseDown — checked BEFORE default handling
void PluginEditor::mouseDown(const juce::MouseEvent& e)
{
    if (windowMode_ != WindowMode::Full)
    {
        if (getOverlayButtonBounds().contains(e.getPosition()))
        {
            applyWindowMode(WindowMode::Full);
            // Sync optionMode_ back to 0
            // proc_.getGamepad().resetOptionMode();  (if method added)
            return;
        }
    }
    // ... existing arp step grid hit-test code ...
}
```

### Pattern 4: Header Buttons — Drawn in paint()

Per the CONTEXT.md decision, the Mini and Maxi buttons are 20×20px at far-right of the 28px header bar. These can be drawn directly in `paint()` with hit-testing in `mouseDown`. This avoids adding two `TextButton` children whose visibility would need to be managed across mode transitions.

Alternatively they can be `juce::TextButton` members added to the component tree and always visible during Full mode only. The draw-in-paint approach is simpler for small pixel-art icon buttons.

```cpp

// header is getLocalBounds().removeFromTop(28)
if (windowMode_ == WindowMode::Full)
{
    // ◱ Mini button
    const auto miniBtnR = juce::Rectangle<int>(header.getRight()-44, header.getY()+4, 20, 20);
    // ⛶ Maxi button
    const auto maxiBtnR = juce::Rectangle<int>(header.getRight()-22, header.getY()+4, 20, 20);

    g.setColour(Clr::textDim.withAlpha(isHoverMini_ ? 0.9f : 0.4f));
    g.drawText(juce::String::charToString(0x25F1), miniBtnR, juce::Justification::centred);
    g.setColour(Clr::textDim.withAlpha(isHoverMaxi_ ? 0.9f : 0.4f));
    g.drawText(juce::String::charToString(0x2BD0), maxiBtnR, juce::Justification::centred);
}
```

Hit-testing in `mouseDown`:
```cpp
if (windowMode_ == WindowMode::Full)
{
    const auto miniBtnR = juce::Rectangle<int>(...);
    const auto maxiBtnR = juce::Rectangle<int>(...);
    if (miniBtnR.contains(e.getPosition())) { applyWindowMode(WindowMode::Mini); return; }
    if (maxiBtnR.contains(e.getPosition())) { applyWindowMode(WindowMode::Maxi); return; }
}
```

Hover state for button brightness: add `bool isHoverMini_, isHoverMaxi_` members, update in `mouseMove` and `mouseExit`, call `repaint(header)`.

### Pattern 5: setAllNonPadComponentsVisible Helper

The component list is large. Build an internal helper that iterates all children except `joystickPad_`:

```cpp
void PluginEditor::setAllNonPadComponentsVisible(bool visible)
{
    for (auto* child : getChildren())
    {
        if (child != &joystickPad_)
            child->setVisible(visible);
    }
}
```

**Important caveat:** Some components are already conditionally hidden in Full mode (e.g. `filterModHintLabel_` at `{0,0,0,0}`, `singleChanTargetBox_` vs `voiceChBox_`). The `setAllNonPadComponentsVisible(true)` call should NOT blindly show everything — it should restore visibility to the Full-mode state. Options:

A. Call `applyWindowMode(Full)` which calls `setSize` which triggers `resized()` which re-applies all Full-mode visibility logic. This is the correct approach — `resized()` already manages conditional visibility.

B. Iterate children and only call `setVisible(false)` when going to Mini/Maxi; on returning to Full call `resized()` directly (but `resized()` is normally called by `setSize`, not called manually).

**Chosen approach (from CONTEXT.md):** `applyWindowMode` calls `setSize(targetW, targetH)` → triggers `resized()` → Full layout code re-runs and correctly restores all component visibility. The `setAllNonPadComponentsVisible(false)` is called BEFORE `setSize` when entering Mini/Maxi, then `setSize` (in the Mini/Maxi path) triggers `resized()` which runs the `else` branch (pad only). On return to Full, `setSize` triggers Full branch which restores everything.

### Anti-Patterns to Avoid

- **Do NOT call `resized()` directly** — call `setSize()` instead; JUCE calls `resized()` as a consequence.
- **Do NOT store savedUiScale_ in Mini/Maxi mode** — the `proc_.savedUiScale_.store()` call in `resized()` must be guarded by `windowMode_ == Full`.
- **Do NOT leave aspect ratio constraint active when entering Mini/Maxi** — a square window (1:1 ratio) will be rejected by the 4:3 constrainer. Call `getConstrainer()->setFixedAspectRatio(0.0)` before `setSize` in Mini/Maxi.
- **Do NOT use `juce::AffineTransform`** on the Graphics context for scaling — this breaks mouse hit-testing. All scaling is via bounds arithmetic only.
- **Do NOT add the overlay button as a child component** — it would need to be excluded from the "hide all" pass and added back every mode transition. Draw in `paintOverChildren` instead.
- **Do NOT call `setAllNonPadComponentsVisible(true)` blindly** — some components are already conditionally hidden in Full mode. Let `resized()` (triggered by `setSize`) restore Full layout.

## Common Pitfalls

### Pitfall 1: Aspect Ratio Constraint Blocks setSize

**What goes wrong:** Calling `setSize(padSize, padSize)` while `setFixedAspectRatio(1120.0/840.0)` is active causes JUCE to constrain the size to maintain 4:3 — the square window will be rejected or distorted.

**Why it happens:** `ComponentBoundsConstrainer::checkBounds` enforces the aspect ratio on every resize attempt. `setSize` goes through the constrainer.

**How to avoid:** In `applyWindowMode`, always call `getConstrainer()->setFixedAspectRatio(0.0)` AND `setResizeLimits(w, h, w, h)` BEFORE calling `setSize(w, h)` for Mini/Maxi. Restore `setFixedAspectRatio(1120.0/840.0)` and `setResizeLimits(840, 630, 1120, 840)` when returning to Full.

**Warning signs:** Window snaps to wrong size, or maintains 4:3 when you expect a square.

### Pitfall 2: savedUiScale_ Corrupted by Mini/Maxi Resize

**What goes wrong:** `resized()` currently has `proc_.savedUiScale_.store((double)scaleFactor_)` unconditionally. When entering Mini mode, `getWidth()` is ~308px, so `scaleFactor_ = 308/1120 ≈ 0.275`. This gets stored as the saved scale, and the next time the plugin opens it will start at a tiny size.

**Why it happens:** `resized()` was written assuming it only runs in Full mode.

**How to avoid:** Guard the `scaleFactor_` derivation and `savedUiScale_.store()` inside the `if (windowMode_ == Full)` branch. In Mini/Maxi, `scaleFactor_` should not be recomputed from `getWidth()`.

### Pitfall 3: Divider Lines and Logo Drawn in Full-Mode-Only paint() Code

**What goes wrong:** The divider lines (lines 4717–4718), logo (4732–4750), and footer text (5329–5387) are drawn in `paint()` using `dividerX_`, `dividerX2_`, etc. These variables are only set in the Full branch of `resized()`. If `windowMode_ != Full` and these draw calls run, they will use stale values and draw artifacts on the pad-only window.

**Why it happens:** `paint()` currently has no mode check.

**How to avoid:** Wrap all Full-mode-specific draw calls in `paint()` with `if (windowMode_ == WindowMode::Full)`. The JoystickPad draws itself (it's a child component) — `paint()` on the editor draws only the chrome around it.

### Pitfall 4: mouseDown Arp Grid Code Runs Before Overlay Check

**What goes wrong:** The existing `mouseDown` checks `arpStepRowABounds_.contains(e.getPosition())`. If the overlay button overlaps with stale arp bounds from a previous Full layout, the arp step will be toggled instead of the return-to-Full action firing.

**Why it happens:** `mouseDown` currently has no mode guard.

**How to avoid:** Add the overlay button hit-test as the FIRST check in `mouseDown`, with an early return. Since the arp bounds are only valid in Full mode (where the overlay button is hidden), there is no overlap concern — but the guard order still matters for correctness.

### Pitfall 5: getScreenBounds() Returns Wrong Rect Before Window Is On Screen

**What goes wrong:** `getScreenBounds().getCentre()` for the Maxi display query may return a wrong point if called before the window is visible (e.g., in the constructor).

**Why it happens:** `getScreenBounds()` computes based on parent component hierarchy and OS window position — may not be reliable before the window is shown.

**How to avoid:** Only call `applyWindowMode(Maxi)` from user interaction (button click in header, or `timerCallback` optMode change) — never from the constructor. The window is guaranteed to be on-screen by then.

### Pitfall 6: JoystickPad Timer and Spring Physics Run Against Wrong Bounds

**What goes wrong:** JoystickPad's `timerCallback` (30 Hz) computes cursor positions based on `getWidth()/getHeight()`. If Mini/Maxi changes the bounds without the pad recalculating, spring physics targets will be wrong for a frame.

**Why it happens:** `joystickPad_.setBounds(...)` triggers `resized()` on JoystickPad, which recomputes internal geometry. However, the spring target is computed in `timerCallback` based on `getWidth()/getHeight()` at that tick.

**How to avoid:** This is self-healing — within one timer tick (33ms) after `setBounds`, the JoystickPad will have correct dimensions. No special handling needed, but verify visually that there is no 1-frame position glitch.

## Open Questions

1. **setFixedAspectRatio(0.0) — does passing 0.0 actually disable the constraint in JUCE 8.0.4?**
   - What we know: The JUCE `ComponentBoundsConstrainer` API shows `setFixedAspectRatio` with a ratio parameter. Passing 0.0 is the documented way to disable.
   - What's unclear: Not verified against JUCE 8.0.4 source; could be that the method checks `> 0.0` or requires a different sentinel.
   - Recommendation: Check JUCE source at `modules/juce_gui_basics/components/juce_ComponentBoundsConstrainer.cpp` — the implementation is in the local FetchContent JUCE checkout at `build/_deps/juce-src/`. If 0.0 doesn't work, use a custom `ComponentBoundsConstrainer` subclass that overrides `checkBounds` to be a no-op in Mini/Maxi mode.

2. **Host behavior with rapid setSize calls (entering/exiting mode quickly)**
   - What we know: Ableton and Reaper handle `setSize` from the editor fine.
   - What's unclear: Whether a very rapid sequence (e.g. Full→Mini→Full within one timer tick) could confuse a host's window tracking.
   - Recommendation: Not a real-world concern — user-triggered transitions are slow. Add `if (windowMode_ == newMode) return;` guard in `applyWindowMode` as defensive check.

3. **Unicode glyph rendering for ◱ (U+25F1) and ↩ (U+21A9) in pixelFont**
   - What we know: The pixel font (Monogram/similar) may not contain these code points; JUCE will fall back to the system font for missing glyphs.
   - Recommendation: Test rendering during UAT. If glyphs don't render, fall back to ASCII approximations: `"[]"` for Mini, `"[ ]"` for Maxi, `"<"` or `"X"` for overlay return.
