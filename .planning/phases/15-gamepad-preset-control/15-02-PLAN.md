```cpp
// ── Preset-scroll mode: MIDI Program Change counter ───────────────────────
// Audio-thread-only. Tracks current program number for absolute PC messages.
// Starts at 0 on plugin load. Not persisted in saved state.
int programCounter_ = 0;
```

In PluginProcessor.cpp processBlock(), locate the existing D-pad BPM block:
```cpp
if (gamepad_.consumeDpadUp())    pendingBpmDelta_.fetch_add(1,  std::memory_order_relaxed);
if (gamepad_.consumeDpadDown())  pendingBpmDelta_.fetch_add(-1, std::memory_order_relaxed);
```

Replace this block with logic that checks preset-scroll mode:

```cpp
// D-pad Up/Down: BPM delta (normal) or PC message (preset-scroll mode)
// In preset-scroll mode: consumePcDelta() returns the delta (GamepadInput fires on button-up).
// In normal mode: consumeDpadUp/Down fire on button-down (unchanged).
if (gamepad_.isPresetScrollActive())
{
    const int pcDelta = gamepad_.consumePcDelta();
    if (pcDelta != 0)
    {
        const int newProgram = juce::jlimit(0, 127, programCounter_ + pcDelta);
        if (newProgram != programCounter_)  // boundary clamp: skip send if already at limit
        {
            programCounter_ = newProgram;
            const int fCh = static_cast<int>(
                apvts.getRawParameterValue(ParamID::filterMidiCh)->load());  // 1-based
            midi.addEvent(
                juce::MidiMessage::programChange(fCh, programCounter_),
                0);  // sample offset 0 within the block
        }
    }
}
else
{
    if (gamepad_.consumeDpadUp())    pendingBpmDelta_.fetch_add(1,  std::memory_order_relaxed);
    if (gamepad_.consumeDpadDown())  pendingBpmDelta_.fetch_add(-1, std::memory_order_relaxed);
}
```

Note: juce::MidiMessage::programChange(channel, programNumber) takes a 1-based channel (1-16) and a 0-based program number (0-127). ParamID::filterMidiCh stores 1-16 directly. No conversion needed.

Note: midi.addEvent() adds to the outgoing MidiBuffer that processBlock() returns to the host. The existing pattern in this function uses this same buffer for note-on/off messages — follow that same pattern. Do NOT use midi.clear() before adding.

Note: programCounter_ is audio-thread-only (never read from message thread). No atomic needed.

In PluginEditor.cpp timerCallback(), locate the section that updates gamepadStatusLabel_. Currently it only updates on connection change events (via onConnectionChangeUI lambda) — the label is not updated in the polling loop.

Add a new block in timerCallback() AFTER the BPM display update and BEFORE or AFTER the filter CC update (anywhere in the polling section is fine):

```cpp
// Update gamepad OPTION indicator for preset-scroll mode
{
    const bool presetScroll = proc_.getGamepad().isPresetScrollActive();
    const bool connected    = proc_.getGamepad().isConnected();

    if (presetScroll)
    {
        // Show OPTION mode indicator — append to existing controller name text
        // Build suffix: existing label prefix (controller name) + " | OPTION"
        // Read existing text to preserve controller name; replace only the color.
        const juce::String currentText = gamepadStatusLabel_.getText();
        // Strip any prior " | OPTION" suffix to avoid accumulation
        const juce::String base = currentText.replace(" | OPTION", "");
        gamepadStatusLabel_.setText(base + " | OPTION", juce::dontSendNotification);
        gamepadStatusLabel_.setColour(juce::Label::textColourId, Clr::highlight);
    }
    else
    {
        // Restore normal color; strip OPTION suffix if present
        const juce::String currentText = gamepadStatusLabel_.getText();
        if (currentText.contains(" | OPTION"))
        {
            const juce::String base = currentText.replace(" | OPTION", "");
            gamepadStatusLabel_.setText(base, juce::dontSendNotification);
        }
        // Restore color based on connection state
        gamepadStatusLabel_.setColour(juce::Label::textColourId,
            connected ? Clr::gateOn : Clr::textDim);
    }
}
```

This approach appends " | OPTION" to whatever the label currently shows (e.g., "PS4 Controller | OPTION") and strips it cleanly on deactivation. The highlight color matches the active/highlight convention used throughout the UI (Clr::highlight is the same color used for LFO buttons, BPM knob fill, etc.).

If `Clr::highlight` is not accessible from timerCallback() scope, use the same color token already used in the file — grep the existing uses of Clr::highlight in PluginEditor.cpp and use the exact same reference. The Clr namespace is defined in PluginEditor.cpp (or included header) and is available throughout the file.

Design constraint from CONTEXT.md: "Subtle — secondary visual weight. Not blinking or animating." The color change + text suffix satisfies this without any animation. The gamepadStatusLabel_ is already placed in the bottom-right of the UI (getWidth() - 8 - 200, getHeight() - 20, 200, 16), which is the gamepad controls section.

No new UI widgets needed. No new APVTS parameters. No changes to PluginEditor.h (no new members required — timerCallback() reads through existing proc_.getGamepad() reference).
