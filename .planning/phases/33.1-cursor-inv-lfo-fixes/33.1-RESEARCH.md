## Summary

All 7 bugs have been traced to their exact source lines. Bugs 1–6 are pure code fixes with no
dependency on external libraries. Bug 7 requires a design decision: the SDL2 2.30.9 API
(`SDL_JoystickCurrentPowerLevel`) maps directly to the existing display's switch cases
(UNKNOWN=-1, EMPTY=0, LOW=1, MEDIUM=2, FULL=3, WIRED=4), and no newer battery API exists in
SDL2 (SDL3-only). The confirmed fallback strategy is: trust EMPTY and LOW states only; treat
FULL as unreliable and display "?" for it when a PS4/PS5 controller is detected via Bluetooth.

**Primary recommendation:** All 7 fixes are small, isolated, and safe to implement in a single
wave with no architectural changes required.

## Bug 1 — Cursor initial position

### Problem
`displayCx_` and `displayCy_` in `JoystickPad` are initialized to `0.0f` (line 163–164 of
`Source/PluginEditor.h`). The first-frame snap guard in `timerCallback()` only fires when
**both** velocity and position are exactly zero:

```cpp
// Source/PluginEditor.cpp line 1128-1132
if (springVelX_ == 0.0f && springVelY_ == 0.0f &&
    displayCx_  == 0.0f && displayCy_  == 0.0f)
{
    displayCx_ = cx;
    displayCy_ = cy;
}
```

The snap fires on the **first** `timerCallback()` tick (60 Hz timer, starts immediately in
`JoystickPad` constructor, line 906). However, the initial `cx/cy` computed in
`timerCallback()` depend on `proc_.joystickX/Y` which default to `0.0f`. The pad center in
pixel space is `w * 0.5f`, not `0.0f`. So `displayCx_ = 0.0f` renders the cursor at the
left edge rather than center.

### Root cause
The first-frame snap sets `displayCx_ = cx` where `cx` is derived from joystick values that
default to 0. The mapping from joystick `[-1, +1]` to pixel space places 0 at the pad
center. So `cx` is correctly the pad center when joystick is at 0 — **but only after
`getWidth()` and `getHeight()` are non-zero**, which happens after `resized()` is called.
The 60 Hz timer may fire before `resized()` has been called, making `w=0, h=0`, causing the
early-return at line 1067 (`if (w < 1.0f || h < 1.0f) return;`) to skip the snap
assignment. The second tick fires with valid dimensions but the snap guard has already been
"used up" (velocity is still 0 and position is still 0 if the early-return fired), so it
snaps to the correct center. Actually the snap fires correctly — the issue is that
`displayCx_/Cy_` are `0.0f` but the **pad center in pixel space** depends on pad width.

### Confirmed fix
In `JoystickPad::resized()` (line 909), after baking `spaceBgBaked_`, reset the display
position to the pad center and clear spring velocity:

```cpp
// Add at end of JoystickPad::resized(), after spaceBgBaked_ is baked
// Snap cursor to pad center so it doesn't appear at (0,0) until the first
// timerCallback fires with a valid size.
{
    const float cx0 = getWidth()  * 0.5f;
    const float cy0 = getHeight() * 0.5f;
    displayCx_  = cx0;
    displayCy_  = cy0;
    springVelX_ = 0.0f;
    springVelY_ = 0.0f;
}
```

**Files:** `Source/PluginEditor.cpp` — `JoystickPad::resized()` starting at line 909.

**Note:** Setting `displayCx_/Cy_` to the pad center (half-width, half-height) is correct
because the spring target `cx` in `timerCallback()` is computed as `w * 0.5f + joystickX *
(w * 0.5f)`, giving the center when joystick is 0. Clearing velocities prevents a spring
artifact on resize.

## Bug 2 — Spring-back too bouncy

### Problem
The damping constant is `0.72f` (line 1125 of `Source/PluginEditor.cpp`):

```cpp
// Source/PluginEditor.cpp line 1119-1142
// kSpring=0.18, kDamping=0.72 → ~90ms settle at 60Hz, slight underdamp overshoot.
{
    constexpr float kSpring  = 0.18f;
    constexpr float kDamping = 0.72f;
    ...
    springVelX_ += (cx - displayCx_) * kSpring;
    springVelY_ += (cy - displayCy_) * kSpring;
    springVelX_ *= kDamping;
    springVelY_ *= kDamping;
    displayCx_  += springVelX_;
    displayCy_  += springVelY_;
}
```

With `kSpring=0.18` and `kDamping=0.72`, the system is underdamped and overshoots.

### Confirmed fix
Change `kDamping` from `0.72f` to `0.90f`:

```cpp
constexpr float kDamping = 0.90f;
```

At `kSpring=0.18`, `kDamping=0.90f` is approximately critically damped to slightly
overdamped — settles without oscillation. The comment on line 1120 should also be updated.

**File:** `Source/PluginEditor.cpp` line 1125.

## Bug 3 — LFO rate modulation asymmetric

### Problem
The parameter `lfoXRate` / `lfoYRate` uses a skewed `NormalisableRange`:

```cpp
// Source/PluginProcessor.cpp lines 295-302
layout.add(std::make_unique<juce::AudioParameterFloat>(
    ParamID::lfoXRate, "LFO X Rate",
    juce::NormalisableRange<float>(0.01f, 20.0f, 0.0f, 0.35f),
    1.0f));
```

The skew factor `0.35f` means the parameter is stored as a **normalized** value
`[0, 1]` that maps non-linearly to Hz via `convertFrom01`. The raw float value returned
by `getRawParameterValue(ParamID::lfoXRate)->load()` is the **denormalized Hz value**
(0.01–20.0). The current fix adds a linear offset to this Hz value:

```cpp
// Source/PluginProcessor.cpp lines 1865-1869
const float base   = apvts.getRawParameterValue(ParamID::lfoXRate)->load();
const float actual = juce::jlimit(0.01f, 20.0f, base + stick * 9.995f);
lfoXRateDisplay_.store(actual, std::memory_order_relaxed);
lfoXRateOverride_ = actual;
```

Because the Hz scale is logarithmic (skew 0.35), adding `stick * 9.995f` in linear Hz
space causes massive asymmetry: moving the stick up from 1 Hz adds up to +9.995 Hz (toward
20 Hz), but moving it down from 1 Hz subtracts only at most 0.99 Hz (clamped at 0.01 Hz).
The perceived modulation depth is wildly different in each direction.

### Confirmed fix — normalized log space

The JUCE `NormalisableRange` with a skew factor converts between Hz and a normalized
`[0, 1]` value via its `convertTo0to1` and `convertFrom0to1` methods. `getRawParameterValue`
returns the Hz value directly. The correct approach is to convert the Hz base to normalized
space, add the stick offset there, then convert back:

```cpp
// Source/PluginProcessor.cpp — replace lines 1865-1869
{
    const float base   = apvts.getRawParameterValue(ParamID::lfoXRate)->load();
    // Convert to normalized [0,1] log space, apply stick offset, convert back
    static const juce::NormalisableRange<float> kLfoRange(0.01f, 20.0f, 0.0f, 0.35f);
    const float norm   = kLfoRange.convertTo0to1(base);
    const float actual = kLfoRange.convertFrom0to1(juce::jlimit(0.0f, 1.0f, norm + stick * 0.5f));
    lfoXRateDisplay_.store(actual, std::memory_order_relaxed);
    lfoXRateOverride_ = actual;
}
```

And the same pattern for `lfoYRate` (lines ~1916-1920):

```cpp
{
    const float base   = apvts.getRawParameterValue(ParamID::lfoYRate)->load();
    static const juce::NormalisableRange<float> kLfoRange(0.01f, 20.0f, 0.0f, 0.35f);
    const float norm   = kLfoRange.convertTo0to1(base);
    const float actual = kLfoRange.convertFrom0to1(juce::jlimit(0.0f, 1.0f, norm + stick * 0.5f));
    lfoYRateDisplay_.store(actual, std::memory_order_relaxed);
    lfoYRateOverride_ = actual;
}
```

**Key note on `stick * 0.5f`:** With normalization, `stick` ranges from -1 to +1 after
attenuation. A scale of `0.5f` gives ±50% of the normalized range (half-range in log space)
at full stick deflection. This is symmetric and musically useful. The planner should
document that the scale factor `0.5f` is a tunable constant.

**Files:** `Source/PluginProcessor.cpp` lines 1865–1869 (xMode case 4) and ~1916–1920
(yMode case 4).

## Bug 6 — Piano black key misses

### Problem
`ScaleKeyboard::mouseDown()` iterates notes 0–11 in forward order:

```cpp
// Source/PluginEditor.cpp lines 2001-2046
void ScaleKeyboard::mouseDown(const juce::MouseEvent& e)
{
    for (int n = 0; n < 12; ++n)
    {
        if (!noteRect(n).contains(e.position))
            continue;
        // ... toggle note
        return;
    }
}
```

White key rects overlap with the lower portion of black key rects (this is how piano layouts
work — white keys have a full-height rectangle, black keys are narrower and shorter but
positioned on top). The iteration order 0..11 checks C (white), C# (black), D (white), etc.
For a click in the black key region of C#, `noteRect(0)` (C white) may match first because
the white key rect extends under the black key area, causing the wrong note to toggle.

The `paint()` method already handles this correctly with a two-pass draw (white keys first,
black keys on top, lines 2054–2079), but `mouseDown()` uses a single forward pass.

### `kIsBlack` array (line 1893):
```cpp
const bool ScaleKeyboard::kIsBlack[12] = {0,1,0,1,0,0,1,0,1,0,1,0};
// Notes:                                  C C# D D# E F F# G G# A A# B
// Black:                                     1    1       1    1    1
```

### Confirmed fix — two-pass hit test

```cpp
void ScaleKeyboard::mouseDown(const juce::MouseEvent& e)
{
    // Two-pass hit test: black keys first (drawn on top), then white keys.
    // This prevents white key rects from stealing clicks meant for black keys.
    static const int kBlackNotes[] = {1, 3, 6, 8, 10};
    static const int kWhiteNotes[] = {0, 2, 4, 5, 7, 9, 11};

    auto tryToggle = [&](int n) -> bool
    {
        if (!noteRect(n).contains(e.position)) return false;
        // ... [rest of existing toggle logic, unchanged]
        return true;
    };

    for (int n : kBlackNotes) { if (tryToggle(n)) return; }
    for (int n : kWhiteNotes) { if (tryToggle(n)) return; }
}
```

The toggle logic body (lines 2008–2044 of the current single loop) is factored into the
lambda unchanged. Only the iteration order changes.

**File:** `Source/PluginEditor.cpp` — `ScaleKeyboard::mouseDown()` at line 2001.

## Bug 7 — Battery level wrong

### SDL2 2.30.9 battery API — confirmed findings

**SDL_JoystickPowerLevel enum values** (from `build/_deps/sdl2-src/include/SDL_joystick.h`
lines 104–111):

```cpp
typedef enum
{
    SDL_JOYSTICK_POWER_UNKNOWN = -1,
    SDL_JOYSTICK_POWER_EMPTY,   /* <= 5%  */  // = 0
    SDL_JOYSTICK_POWER_LOW,     /* <= 20% */  // = 1
    SDL_JOYSTICK_POWER_MEDIUM,  /* <= 70% */  // = 2
    SDL_JOYSTICK_POWER_FULL,    /* <= 100% */ // = 3
    SDL_JOYSTICK_POWER_WIRED,                 // = 4
    SDL_JOYSTICK_POWER_MAX                    // = 5
} SDL_JoystickPowerLevel;
```

These map directly to `getBatteryLevel()` return value and the existing switch in
`PluginEditor::paint()` (lines 4034–4041):
- `-2` = no controller (not from SDL, set by `getBatteryLevel()` null guard)
- `-1` = UNKNOWN → shows `?`
- `0` = EMPTY → red
- `1` = LOW → orange
- `2` = MEDIUM → yellow
- `3` = FULL → green
- `4` = WIRED → cyan

**No improved SDL2 API exists.** SDL3 has `SDL_GetJoystickPowerInfo()` which returns
percentage + charging state, but this project uses SDL2 2.30.9. There is no SDL2 equivalent.

**Known platform behavior:** SDL2's HIDAPI driver for PS4/PS5 on Windows via Bluetooth
frequently reports `SDL_JOYSTICK_POWER_FULL` even when the controller battery is low.
This is a known SDL2 limitation — the HIDAPI report for battery data requires the controller
to push a battery status report, which PS4/PS5 controllers do infrequently or not at all
in SDL2's polling model on Windows. SDL2 does NOT query battery percentage on demand.

**Confirmed fallback strategy:**

Change `GamepadInput::getBatteryLevel()` to return `SDL_JOYSTICK_POWER_UNKNOWN` (-1) when
the raw SDL result is `SDL_JOYSTICK_POWER_FULL` (3) AND the controller is connected via
Bluetooth (not wired). This turns the full-bar display into a `?` for Bluetooth PS controllers
while still showing the correct icon for EMPTY and LOW (which SDL does report reliably when
the battery is critically depleted).

```cpp
// Source/GamepadInput.cpp — replace lines 399-405
int GamepadInput::getBatteryLevel() const
{
    if (!controller_) return -2;
    SDL_Joystick* joy = SDL_GameControllerGetJoystick(controller_);
    if (!joy) return -2;
    const SDL_JoystickPowerLevel raw = SDL_JoystickCurrentPowerLevel(joy);
    // SDL_JOYSTICK_POWER_FULL is unreliable on Windows Bluetooth PS4/PS5 —
    // often returns FULL even at low battery. Return UNKNOWN so UI shows "?".
    // EMPTY and LOW are reliable (controller signals these at critical depletion).
    // WIRED is reliable (USB detection is OS-level, not controller-reported).
    if (raw == SDL_JOYSTICK_POWER_FULL)
        return (int)SDL_JOYSTICK_POWER_UNKNOWN;  // → shows "?" in UI
    return (int)raw;
}
```

**Alternative (simpler):** Always return UNKNOWN for non-critical states and only show
EMPTY/LOW/WIRED:

```cpp
int GamepadInput::getBatteryLevel() const
{
    if (!controller_) return -2;
    SDL_Joystick* joy = SDL_GameControllerGetJoystick(controller_);
    if (!joy) return -2;
    const SDL_JoystickPowerLevel raw = SDL_JoystickCurrentPowerLevel(joy);
    // Only trust EMPTY, LOW, and WIRED. MEDIUM and FULL are unreliable on Windows.
    if (raw == SDL_JOYSTICK_POWER_EMPTY || raw == SDL_JOYSTICK_POWER_LOW ||
        raw == SDL_JOYSTICK_POWER_WIRED)
        return (int)raw;
    return (int)SDL_JOYSTICK_POWER_UNKNOWN;  // → shows "?"
}
```

**Recommended:** The "only EMPTY/LOW/WIRED" approach (simpler, handles MEDIUM unreliability
too). The UI already has the `?` rendering path at line 4062 for `batteryLevel_ == -1`.

**No UI changes needed** — the existing `?` path at line 4062 already covers this case.

**Files:** `Source/GamepadInput.cpp` lines 399–405 only.

## Architecture Patterns

### JUCE APVTS Attachment Detach/Reattach
JUCE's `AudioProcessorValueTreeState::SliderAttachment` and `ComboBoxAttachment` are RAII.
Calling `reset()` on the `unique_ptr` destroys the attachment, which removes the parameter
listener registration in its destructor (JUCE source: `ParameterAttachment::removeListener`).
Creating a new attachment with `make_unique` with a different param ID re-registers. This is
the documented safe pattern and runs on the message thread.

**Caution:** Do not call `reset()` / `make_unique` from inside a parameter callback or audio
thread. Only safe from timer callback or UI event handlers (message thread).

### NormalisableRange Log-Space Arithmetic
For `NormalisableRange(min, max, step, skew)`:
- `convertTo0to1(hz)` — returns the normalized position [0,1] on the skewed curve
- `convertFrom0to1(norm)` — returns the Hz value for a given normalized position

Adding offsets in normalized space gives perceptually uniform modulation because the
normalized space is already uniform on the skewed scale. This is the correct idiom for
log-scaled parameters.

### Spring-Damper Tuning (JUCE 60 Hz timer)
The discrete-time spring equation used here is:
```
vel += (target - pos) * kSpring
vel *= kDamping
pos += vel
```
At 60 Hz, `kSpring=0.18`, `kDamping=0.90f` gives critical/overdamped response (~150ms
settle, no overshoot). `kDamping=0.72f` is underdamped at these spring values (overshoots).

## Common Pitfalls

### Pitfall 1 — APVTS attachment swap on message thread only
**What goes wrong:** If reattach runs on the audio thread or parameter callback, JUCE
asserts. The `timerCallback()` runs on the message thread — safe.

### Pitfall 2 — Resetting attachment drops current param value to default
**What goes wrong:** Destroying and recreating an attachment may snap the UI control to
the param's current value (which is correct) but may also snap to 0 if the attachment
reads before APVTS sync. In practice this works correctly because JUCE's attachment
constructor reads `getRawParameterValue()` and sets the control immediately.

### Pitfall 3 — Two-pass mouseDown: lambda captures
**What goes wrong:** Factoring the toggle body into a lambda that captures `[&]` and returns
`bool` works correctly in C++17. The existing `return;` in the body becomes `return true;`
in the lambda.

### Pitfall 4 — NormalisableRange in processBlock
**What goes wrong:** Constructing `NormalisableRange` objects in `processBlock` with `static`
is technically fine (initialized once) but the range constants must match the APVTS definition
exactly or modulation will be miscalibrated. Use `static const` or `constexpr` with the same
values as the APVTS layout.
